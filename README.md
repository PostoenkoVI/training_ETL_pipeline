# training_ETL_pipeline

ETL-пайплайн для загрузки данных из разнородных источников (CSV, JSON, XML) в PostgreSQL с построением хранилища данных по методологии **Data Vault 2.0** и выполнением аналитических витрин.

---


---

##  Установка и запуск

### Через Docker Compose

1. Установить **Docker** и **Docker Compose**.
2. Клонировать репозиторий:
   ```bash
   git clone https://github.com/PostoenkoVI/training_ETL_pipeline.git
   cd training_ETL_pipeline
   ```
3. ```bash
   docker-compose up --build
   ```
  Поднимется контейнер postgres_db с PostgreSQL 15. Затем запустится контейнер test_app, который выполнит main.py. Для последующих загрузок данных (при замене файлов в папке `./data`) и дополнительных опций генерации аналитических запросов рекомендуется использовать опции:
   ```bash
   docker-compose up app [--options]
   ```
4. Совместимость различных типов входных данных кроме предоставленных изначально может быть добавлена в будущих обновлениях.

## Архитектурные решения

  В проекте реализована схема хранения данных Data Vault 2.0. Для ее имплементации все таблицы сначала читаются в схему raw, затем дедублецируются, форматируются, очищаются от проблемных записей путем логгирования их в схему logs, очищенные данные собираются в схему cleaned, из которой далее в схеме dwh собираются хабы и спутники для каждой изначальной таблицы (покупатели, продукты, события, платежи и заказы) и три таблицы-ссылки (платеж-заказ, событие-покупатель-продукт, заказ-покупатель-продукт). Ссылки не разделяются из-за жестко заданного формата входных данных. После этого на базе dwh строятся аналитические запросы. 
  
  В связи с качеством первичных данных (систематическое несовпадение стоимостей продуктов с одинаковыми id в таблицах orders и products и другими) id продукта, стоимость и тд и аналогичные части считаются свойствами заказа, а не свойствами продукта (т.е. остаются как доп свойства в соответствующих таблицах сателлитах, т.к. содержат новую информацию). Иначе, если их считать несоответствиями требуется установить приоритет истинности и логгировать как проблемные порядка 95-100% датасета. Общая сумма покупок считается по таблице payments, для подсчета популярности добавлены опции в функции генерации аналитики.

  По умолчанию результат на входных данных воспроизводится при создании контейнера, таблица логов работает по правилу INSERT ONLY. В экспериментальных целях добавлена опция очистки базы данных в начале пайплайна. В целях экономии места при масштабировании raw и cleaned схемы удаляются после генерации dwh схемы.

## Опции командной строки
  ## ⚙️ Параметры командной строки

| Параметр | Описание | Значение по умолчанию |
|----------|----------|----------------------|
| `--reset-db` | Пересоздать все схемы (raw, cleaned, logs, dwh) | `False` |
| `--products-path` | Путь к файлу товаров (Excel) | `/app_data/products.xlsx` |
| `--customers-path` | Путь к файлу клиентов (CSV) | `/app_data/customers.csv` |
| `--payments-path` | Путь к файлу платежей (CSV, разделитель `^`) | `/app_data/payments.csv` |
| `--orders-path` | Путь к файлу заказов (JSON) | `/app_data/orders.json` |
| `--events-path` | Путь к файлу событий (XML) | `/app_data/events.xml` |
| `--prefered_currency` | Предпочтительная валюта для аналитики | `RUB` |
| `--rub`, `--usd`, `--eur` | Курсы валют (относительно предпочтительной) | `1`, `75`, `85` |
| `--top_products_by` | Критерий для топ-товаров (`views`, `clicks`, `purchases`, `orders`) | `views` |
| `--top_customers_last_date_by` | Критерий для топ-клиентов (`orders`, `purchases`) | `orders` |

## Структура проекта

```
training_ETL_pipeline/
├── data/ # Входные данные
│ ├── customers.csv # Клиенты
│ ├── products.csv # Товары
│ ├── payments.csv # Платежи (разделитель ^)
│ ├── orders.json # Заказы (массив объектов)
│ ├── events.xml # События (вложенный XML)
│ └── output/ # Результаты работы пайплайна (создаётся автоматически)
│ ├── structure_example_.txt # Выгрузка всех таблиц (примеры + количество записей)
│ └── _.csv # Результаты аналитических запросов
│
├── sql/ # SQL-скрипты
│ ├── init_raw_cleaned_tables.sql # Создание схем raw, cleaned, logs + таблиц
│ ├── reset_schemas.sql # Сброс всех схем (raw, cleaned, logs, dwh)
│ ├── drop_public_functions.sql # Удаление функций из public
│ ├── read_.sql # Скрипты трансформации raw → cleaned
│ ├── update_hubs.sql # Обновление хабов Data Vault
│ ├── sat.sql # Сателлиты для каждой сущности
│ ├── link.sql # Связи (линки) между сущностями
│ └── analytics_*.sql # Аналитические функции (витрины)
│
├── main.py # Основной скрипт ETL
├── requirements.txt # Зависимости Python
├── Dockerfile # Образ для приложения
├── docker-compose.yml # Оркестрация: Postgres + приложение
├── .dockerignore # Исключения для Docker
└── README.md # Этот файл
```
